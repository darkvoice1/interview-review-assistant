# 中文 Commit 模板（详细分点版）

> 使用方式：复制下方“模板正文”，把尖括号占位符替换成你的实际内容。

## 标题格式

`<type>(<scope>): <一句话总结本次改动>`

- `type` 常用值：`feat` / `fix` / `refactor` / `test` / `docs` / `chore`
- `scope` 常用值：`oauth` / `wecom` / `auth-router` / `storage` / `verification` 等
- 标题建议：聚焦单一主题，长度控制在 20~40 字

## 模板正文

```text
<type>(<scope>): <一句话总结本次改动>

改动背景：
1. <为什么要做这次改动（问题/目标）>
2. <当前的限制或风险>

本次改动：
1. <改动点 A（做了什么）>
2. <改动点 B（做了什么）>
3. <改动点 C（做了什么）>

测试补充：
1. <新增或修改了哪些测试>
2. <覆盖了哪些关键场景>
3. <执行结果：通过/失败 + 关键命令>

收益说明：
1. <业务收益>
2. <工程收益（可维护性/可扩展性/稳定性）>
3. <后续可复用价值>
```

## 示例（可参考）

```text
feat(wecom): 实现企业微信 OAuth 回调主链路并补齐单测

改动背景：
1. 现有 WeCom 回调仍为占位实现，无法完成真实登录闭环。
2. OAuth 回调需要在安全校验后完成身份匹配、绑定与建号。

本次改动：
1. 实现 WeCom 回调完整流程：code/state 校验、用户信息获取、身份落库。
2. 优先命中“已绑定身份”，未命中时按邮箱/手机号尝试自动绑定。
3. 增加绑定冲突处理：邮箱与手机号命中不同用户时返回冲突错误。

测试补充：
1. 新增 WeCom Provider 单元测试文件。
2. 覆盖首登建号、已绑定登录、邮箱绑定、绑定冲突、state 无效等场景。
3. 执行相关单测与 typecheck，结果均通过。

收益说明：
1. WeCom OAuth 从占位逻辑升级为可运行主链路。
2. 账号统一策略落地，降低重复账号与误绑定风险。
3. 为 Feishu/Wechat 按同模式接入提供可复用模板。
```
